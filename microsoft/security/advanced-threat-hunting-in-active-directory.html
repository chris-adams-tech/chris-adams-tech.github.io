Advanced Threat Hunting in Active Directory

AD Structure

  • Forest one or more AD domains
  • Domains group of objects
  • Organizational Units containers storing objects

Global Catalog - repository containing searchable representation of every object

AD Domain Service

  1. Stores directory data
  2. Manages domain interactions
  3. Checks credentials and grants access

AD Roles

Flexible Single Master Operation Roles

Forest roles:

  • Schema role - set of rules that classifies objects and attributes; only one domain controller holds this role
  • Domain naming mastering role - ensures every name is unique and registered; one per forest

AD roles:

  • RID master role
  • PDC emulator role - password changes, time synchronization
  • Infrastructure

AD Security Model

  • Uses Kerberos for authenticating users

Roles are managed through:

  • Security principals
    • assigns to users, groups and computer accounts
  • Group Policy
    • sets policy across users and computers
  • Role-based access control
    • regulates access based on assigned roles
    • users that have specific permissions
    • ensures principal of least privilege

Threat Hunting

Getting started:

  1. Formulate a hypothesis.
  2. Gather data.
  3. Analyze the data.
  4. Think as an attacker.
  5. Act on threats.

Threat Hunting in Active Directory

First enable auditing

In the domain controller, go to Group Policy Management > Computer Configuration > Policies > Windows Settings > Security Settings > Advanced Audit Policy Configuration

Choose to enable your specified logs. Note, that if you have everything enabled and uploading to a SIEM, this will generate a lot of data. Ensure your indexer has enough throughput.

To finish enabling the auditing, go to Active Directory Users and Computer under the Manage tab.

  • Ensure that Advanced Features are enabled under the View tab.

Then, select a user and click Properties > Security > Advanced > Auditing

  1. Add a new Auditing entry
  2. Select the Principal as Everyone
  3. Configure the permissions to audit success and failed attempts to read, write, and modify object

Relevant Event IDs

4625 - failed log-on attempts

4672 - Special privileges assigned to new log-on

4732 & 4733 - user added or removed from group

Analyzing AD Logs and Events

On a domain controller, go to Event Viewer. Under the Applications and Service Logs, navigate to Directory Service.

Uncover Lateral Movement and Privilege Escalation

Lateral Movement

  1. Check source network address or workstation name
    1. verify it is authorized
  2. Review recent activity for the admin account
  3. Investigate any unusual activity involving lsass.exe process
  4. Analyze network traffic for unusual patterns

Privilege Escalation

  1. Monitor for changes in user privileges and modifications to GPOs.

Investigation steps:

  1. Determine if compromised user should have these privileges
  2. Look for other security events associated with account such as event ID 4688
  3. Verify if account was recently added to any privileged groups like administrators or domain admins
  4. Analyze network traffic for unusual patterns

Kerberos Attack techniques

Script to detect Kerberroastable accounts:

$kerberoastableAccountsWithPasswordInfo = $kerberoastableAccounts
| ForEach-Object {
		[PSCustomObject]@{
				Name = $_.Name
				SamAccountName = $_.SameAccountName
				ServicePrincipalName = $_.ServicePrincipalName
				Description = $_.Description
				PasswordNeverExpires = PasswordNeverExpires($_.UserAccountControl)
				PasswordLastSet = ($_.PasswordLastSet
				ReversibleEncryption = IsReversibleEncryptionEnabled($_.UserAccountControl)
		}
	}

$kerberoastableAccountsWithPasswordInfo | Format-Table -AutoSize

$kerberoastableAccountsWithPasswordInfo | Export-Csv -Path "KerberoastableAccountsWithPasswordInfo.csv" -NoTypeInformation

Detecting Malicious GPO activities

Relevant Event IDs:

5136 - GPO modification

4670 - Permissions changed

  • Changes made outside of normal operating hours or unusual modifications from unauthorized accounts
  • New GPO policies that are not expected

If anything unexpected occurs

  • Isolate the affected systems
  • conduct further investigation
