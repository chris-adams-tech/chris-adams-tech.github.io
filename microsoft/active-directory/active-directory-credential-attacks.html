Introduction to Active Directory Attacks: DCsync, Golden Ticket, and Kerberoasting

Active Directory (AD) is the backbone of most enterprise networks, managing user authentication, device access, and permissions across the organization. However, its central role also makes it a prime target for attackers. Once an adversary gains a foothold in a network, AD attacks can provide them with privileged access, persistence, and control over the environment, often without detection for extended periods.

In this post, we’ll explore three of the most well-known AD attack techniques: DCsync, Golden Ticket, and Kerberoasting. These attacks exploit inherent AD functionality to escalate privileges, steal credentials, and even compromise entire networks.

This lab was replicated from a blog post from Wazuh https://wazuh.com/blog/how-to-detect-active-directory-attacks-with-wazuh-part-1-of-2/.

Next, add the local rules into the Wazuh server

Adding the local rules

To configure alerts within the Wazuh SIEM, add the rules in the local_rules.xml configuration file.

The configuration file is located at:

/var/ossec/etc/rules/local_rules.xml

For this lab, I will be using the configuration provided by Wazuh:

<group name="security_event, windows,">
 
  <!-- This rule detects DCSync attacks using windows security event on the domain controller -->
  <rule id="110001" level="12">
    <if_sid>60103</if_sid>
    <field name="win.system.eventID">^4662$</field>
    <field name="win.eventdata.properties" type="pcre2">{1131f6aa-9c07-11d1-f79f-00c04fc2dcd2}|{19195a5b-6da0-11d0-afd3-00c04fd930c9}</field>
    <options>no_full_log</options>
    <description>Directory Service Access. Possible DCSync attack</description>
  </rule>
 
 <!-- This rule ignores Directory Service Access originating from machine accounts containing $ -->
 <rule id="110009" level="0">
    <if_sid>60103</if_sid>
    <field name="win.system.eventID">^4662$</field>
    <field name="win.eventdata.properties" type="pcre2">{1131f6aa-9c07-11d1-f79f-00c04fc2dcd2}|{19195a5b-6da0-11d0-afd3-00c04fd930c9}</field>
    <field name="win.eventdata.SubjectUserName" type="pcre2">\$$</field>
    <options>no_full_log</options>
    <description>Ignore all Directory Service Access that is originated from a machine account containing $</description>
  </rule>
 
  <!-- This rule detects Keberoasting attacks using windows security event on the domain controller -->
  <rule id="110002" level="12">
    <if_sid>60103</if_sid>
    <field name="win.system.eventID">^4769$</field>
    <field name="win.eventdata.TicketOptions" type="pcre2">0x40810000</field>
    <field name="win.eventdata.TicketEncryptionType" type="pcre2">0x17</field>
    <options>no_full_log</options>
    <description>Possible Keberoasting attack</description>
  </rule>
 
  <!-- This rule detects Golden Ticket attacks using windows security events on the domain controller -->
  <rule id="110003" level="12">
    <if_sid>60103</if_sid>
    <field name="win.system.eventID">^4624$</field>
    <field name="win.eventdata.LogonGuid" type="pcre2">{00000000-0000-0000-0000-000000000000}</field>
    <field name="win.eventdata.logonType" type="pcre2">3</field>
    <options>no_full_log</options>
    <description>Possible Golden Ticket attack</description>
  </rule>
 
</group>
Apply the changes
systemctl restart wazuh-manager

DCSync

What is a DCSync attack?

DCsync enables attackers to impersonate Domain Controllers (DCs) and replicate sensitive data like password hashes, effectively allowing them to steal credentials for any user in the domain, including privileged accounts.

Golden Ticket

What is a Golden Ticket attack?

Golden Ticket attacks involve forging authentication tickets to impersonate any user within the domain, giving attackers unrestricted access to network resources for as long as the ticket remains valid.

From the captured NTLM hash of the krbtgt account, we can now use the hash, along with the SID of the account to generate an authentication ticket.

While in the same command prompt, by running mimikatz with the command and credentials obtained, this is the result:

mimikatz # kerberos::golden /domain:wazuhtest.com /sid:S-1-5-21-1860018313-2454207738-2274937249 /rc4:<Hash NTLM of KRBTGT> /user:FakeUser /groups:513,2668 /ptt

Now, we got a golden ticket for our fake user hacker for the local Active Directory domain rabbitlabz.local, awesome!

Let's spawn a cmd shell from mimikatz 

mimikatz.exe # misc::cmd

As you can see, mimikatz was able to spawn a command shell prompt then by running klist we can verify that the ticket is currently loaded into memory.

Kerberoasting

What is Kerberoasting?

Kerberoasting leverages service accounts by requesting and cracking their Kerberos tickets, potentially exposing weakly protected accounts that hold significant privileges.

In order to execute this successfully, there are a few things that need to be done.

Setting the Execution Policy for PowerShell

More than likely, there may be a restriction on executing PowerShell scripts. To change the setting via PowerShell run:

Set-ExecutionPolicy -ExecutionPolicy Unrestricted

Set the SPN for the service account with a vulnerable password

Then, in the kerberoast directory, run the command to get the SPN's.

.\GetUserSPNs.ps1

The service account that was created should appear

The commands below will request an RC4-encrypted Kerberos TGS ticket:

Then, launch mimikatz from PowerShell and run the following:

C:\Users\john-adm\mimikatz_trunk\x64\mimikatz.exe

mimikatz # kerberos::list /export

After this is run, a file will be created in the kerberoast directory
